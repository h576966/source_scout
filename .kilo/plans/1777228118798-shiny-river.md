# Plan: GitHub Project Finder MCP

## Context

The project directory is a bare Kilo Code template — no application code exists yet. The goal is to build a FastMCP-based MCP server that enables AI coding agents to find, evaluate, and extract value from relevant public GitHub repositories. The server is a decision + compression layer on top of GitHub, not a search wrapper.

Key design decisions already locked:
- GITHUB_TOKEN is REQUIRED (server errors on startup if unset)
- Search depth: fetch 30, rank locally, return top 10
- MVP pattern extractor: README + file tree + up to 5 key file previews (first 30 lines each)
- Cache TTLs are configurable via env vars with documented defaults
- All async (httpx + async FastMCP tools)
- Plain functions over classes (follow 00-conventions.md)

FastMCP v3.x research confirms:
- `@mcp.tool()` decorator supports async natively (sync tools run in threadpool)
- Pydantic `BaseModel` params auto-validate and generate JSON schema
- Returning dataclass/Pydantic models auto-generates `structuredContent`
- Error handling: raise exceptions (use `ToolError` for public messages, runtime errors for internal)
- HTTP transport or stdio supported; stdio is default for MCP clients
- Minimum dependency: `fastmcp` (bundles pydantic, starlette, uvicorn)

GitHub REST API research confirms:
- `/search/repositories` supports `per_page` up to 100, `sort` by stars/forks/updated
- Rate limit headers: `x-ratelimit-remaining`, `x-ratelimit-reset`, `x-ratelimit-resource`
- Search-specific rate limit: 30 req/min (auth'd)
- README endpoint returns base64-encoded content
- Contents endpoint returns file/directory listings
- Link header for pagination: `rel="next"` pattern

## Approach

Build the MCP server in 5 phases (P0-P4 + Phase 2), each phase producing independently testable modules. Phase 0 creates project scaffolding (docs, rules, skills, config). Phases 1-3 build the modules bottom-up (client → cache → ranker → inspector → extractor → server). Phase 4 polishes with packaging and final verification. Phase 2 adds local cloning via GitPython, deeper pattern extraction, a `deep_inspect_repo` tool, configurable ranker weights, and a pattern library.

Use DuckDB over SQLite for the cache layer — simpler API, no separate file required if using in-memory mode, supports concurrent reads. Since the cache is local and ephemeral by design, this is a better fit than SQLite for our access patterns.

**github-api/SKILL.md** — GitHub API patterns the LLM needs:
```markdown
# GitHub API Patterns

## Search Endpoint
- URL: `GET https://api.github.com/search/repositories`
- Query: `q` parameter with qualifiers (language:, stars:, topic:, pushed:, archived:, license:, in:readme)
- Sort: `stars`, `forks`, `updated`, `best match`
- Pagination: `per_page` (max 100), `page` (max 10 pages, 1000 total results)
- Rate limit: 30 req/min for auth'd search

## Rate Limit Headers
- `x-ratelimit-remaining`: requests left
- `x-ratelimit-reset`: UTC epoch seconds when window resets
- `x-ratelimit-resource`: `core` or `search`
- Check `x-ratelimit-remaining == 0` to detect exhaustion
- Use `retry-after` header for secondary rate limits

## Readme Endpoint
- URL: `GET https://api.github.com/repos/{owner}/{repo}/readme`
- Response: `{ content: "base64-string", encoding: "base64" }`
- Must decode base64 content
- Accept header `application/vnd.github.raw+json` for raw content

## Contents Endpoint (File Tree)
- URL: `GET https://api.github.com/repos/{owner}/{repo}/contents/{path}`
- Root directory listing when path is empty or `""`
- Returns array for directories, single object for files
- Max 1000 files per directory
- Files >1MB return `encoding: "none"` and empty `content`

## Error Handling
- 403 with `x-ratelimit-remaining: 0` → rate limited, use `x-ratelimit-reset` for retry time
- 404 → repo not found or private
- 422 → validation error (bad query syntax)
```

# FastMCP Patterns

## Tool Definition
```python
from fastmcp import FastMCP
from fastmcp.exceptions import ToolError

mcp = FastMCP("RepoFinder")

@mcp.tool(annotations={"readOnlyHint": True})

- All I/O tools use `async def`
- Use `httpx.AsyncClient` for HTTP calls
- Sync tools auto-run in threadpool (no blocking)

## Input Validation

- Use `Annotated[str, Field(description="...")]` for simple parameter docs
- FastMCP auto-generates JSON schema from type hints

## Output Pattern
- Return dataclass or Pydantic model for auto `structuredContent`
- Always include: `verdict`, `cached`, `timestamp` fields
- For list results, wrap in a container model

## Error Pattern
- `ToolError("message")` for validation/user errors
- `RuntimeError("message")` for system errors
```

### P1-1: GitHub Client

**File:** `src/repo_finder/github_client.py`

**Functions:**

1. `async def search_repos(query: str, per_page: int = 30, sort: str = "stars", order: str = "desc") -> list[dict]`
   - Builds search URL with encoded query
   - Returns list of repo dicts from `items` field
   - Handles rate limits (checks headers, raises RateLimitError)
   - Handles 422 (invalid query) → ToolError
   - Handles 404 (no results) → empty list

2. `async def get_repo_metadata(owner: str, repo: str) -> dict`
   - GET `/repos/{owner}/{repo}`
   - Returns full repo metadata dict
   - Handles 404 → ToolError("Repository not found or is private")
   - Handles rate limits

3. `async def get_readme(owner: str, repo: str) -> str | None`
   - GET `/repos/{owner}/{repo}/readme`
   - Decodes base64 content to string
   - Returns None if no README exists (404)
   - Truncates to 10KB to avoid context bloat

4. `async def get_commits(owner: str, repo: str, per_page: int = 20) -> list[dict]`
   - GET `/repos/{owner}/{repo}/commits?per_page=20`
   - Returns list of commit dicts (sha, date, message, author)

5. `async def get_repo_contents(owner: str, repo: str, path: str = "") -> list[dict]`
   - GET `/repos/{owner}/{repo}/contents/{path}`
   - Returns directory listing or file object
   - For directories: returns array of entries sorted by type (dirs first)
   - For files: returns single object with content (if ≤1MB)

6. `async def get_file_content(owner: str, repo: str, path: str, max_lines: int = 30) -> str | None`
   - GET `/repos/{owner}/{repo}/contents/{path}`
   - Decodes base64, returns first `max_lines` lines
   - Returns None if file doesn't exist or is too large

**Internal helper:**
- `async def _get(url: str) -> httpx.Response` — shared httpx client with GITHUB_TOKEN header, rate-limit checking

**Dependencies:** httpx, os.environ

**Tests:** `tests/test_github_client.py` — mock httpx transport, test rate limit detection, pagination, error cases

### P1-2: Cache Layer

**File:** `src/repo_finder/cache.py`

**Design:** DuckDB in-memory database with optional persistence via `CACHE_DB_PATH` env var. Default is in-memory (cache lost on restart).

**Functions:**

1. `def init_cache() -> duckdb.DuckDBPyConnection`
   - Creates connection, creates `cache` table if not exists
   - Schema: `key TEXT PRIMARY KEY, value JSON, created_at TIMESTAMP, ttl_seconds INTEGER`

2. `async def cache_get(key: str) -> dict | None`
   - Looks up key, checks if expired (`now - created_at > ttl_seconds`)
   - Returns parsed JSON value if valid, None if missing or expired
   - Deletes expired entries on read (lazy eviction)

3. `async def cache_set(key: str, value: dict, ttl_seconds: int) -> None`
   - UPSERTs key with value, current timestamp, and TTL

4. `async def cache_delete(key: str) -> None`

5. `def get_ttl(cache_type: str) -> int`
   - Maps cache_type → env var → default seconds:
     - `"search"` → `CACHE_TTL_SEARCH` → 1800
     - `"metadata"` → `CACHE_TTL_METADATA` → 3600
     - `"readme"` → `CACHE_TTL_README` → 7200

**Key format conventions:**
- Search: `"search:{sha256(query+'|'+sort+'|'+order)[:16]}"`
- Repo metadata: `"repo:{owner}/{repo}"`
- README: `"readme:{owner}/{repo}"`
- File tree: `"tree:{owner}/{repo}:{path}"`

**Tests:** `tests/test_cache.py` — unit tests for TTL expiry, hit/miss, key format, lazy eviction

### P1-3: Ranker

**File:** `src/repo_finder/ranker.py`

**Design:** Pure functions, no I/O. Fully unit-testable with dict fixtures.

**Functions:**

1. `def score_repo(repo: dict, query: str) -> RepoScore`
   - Returns `RepoScore` dataclass: `{ total: float, breakdown: dict[str, float], verdict: str }`
   - Breakdown keys: `relevance`, `activity`, `popularity`, `structure`, `license`
   - Each sub-score is 0.0–1.0, multiplied by weight, summed to `total` (0.0–1.0)
   - Verdict mapping: `total >= 0.7 → "useful"`, `total >= 0.4 → "maybe"`, else `"skip"`

2. `def score_relevance(repo: dict, query: str) -> float`
   - Weight: 0.4
   - Checks: description contains query terms, topics match query keywords, language matches (if in query)
   - Simple term-overlap scoring, not semantic

3. `def score_activity(repo: dict) -> float`
   - Weight: 0.2
   - `pushed_at`: within 1 month = 1.0, 6 months = 0.6, 1 year = 0.3, older = 0.1
   - `archived = True` → 0.0
   - Average of pushed_at recency and commit frequency

4. `def score_popularity(repo: dict) -> float`
   - Weight: 0.15
   - log(stars + 1) / log(max_stars_in_batch + 1) normalized against batch
   - Also factors forks with 0.3 weight within this sub-score
   - Cap at 5000 stars (diminishing returns)

5. `def score_structure(repo: dict) -> float`
   - Weight: 0.15
   - README exists = +0.3, README > 500 chars = +0.3
   - Topics present = +0.2, License present = +0.2
   - Returns 0.0 if can't determine from metadata alone (actual file tree comes later in inspector)

6. `def score_license(repo: dict) -> float`
   - Weight: 0.1
   - Has license = 1.0, no license = 0.2
   - Permissive license (MIT, Apache, BSD) = 1.0, copyleft (GPL) = 0.8, unknown = 0.5

7. `def rank_repos(repos: list[dict], query: str, top_n: int = 10) -> list[dict]`
   - Scores all repos, sorts by total descending, returns top N
   - Attaches `_score` dict to each repo (total + breakdown + verdict)

**Dataclass:**
```python
@dataclass
class RepoScore:
    total: float
    relevance: float
    activity: float
    popularity: float
    structure: float
    license: float
    verdict: str  # "useful" | "maybe" | "skip"
```

**Tests:** `tests/test_ranker.py` — test each sub-scorer with fixture repos, test rank order, test verdict thresholds, test edge cases (archived, no license, no description)

### P2-2: Deep Repo Cloner (`src/repo_finder/cloner.py`)

**Functions:**

1. `def clone_repo(repo_url: str) -> str`
   - Derives deterministic temp dir via `sha256(repo_url)[:12]`
   - Checks if already cloned (path exists + valid git repo + correct remote URL)
   - Clones via `git.Repo.clone_from()` if not present
   - Raises `ToolError` on clone failure
   - Uses `os.path.join(tempfile.gettempdir(), f"repo_finder_{hash}")` for path

2. `def cleanup_clone(repo_path: str) -> None`
   - `shutil.rmtree` with `ignore_errors=True`
   - Called in `finally` blocks of all clone-using tools

3. `def get_directory_tree(path: str, prefix: str = "") -> str`
   - Recursive Unicode tree generator (box-drawing: `├──`, `└──`, `│   `)
   - Skips `.git` directories
   - Sorts entries alphabetically
   - Returns multiline string

Design rationale: Borrowed directly from `mcp-git-ingest` — proven pattern. Deterministic caching avoids re-cloning the same repo. Cleanup guarantees no disk leakage.

### P2-3: Enhanced Pattern Extractor (`src/repo_finder/pattern_extractor.py`)

Extend existing MVP to use cloned repos when available:

**New functions:**

1. `async def extract_patterns_deep(owner: str, repo: str, focus: str | None = None) -> DeepPatternReport`
   - Clones repo via `cloner.clone_repo()`
   - Walks directory tree for full file listing (no 100-file API limit)
   - Reads ALL key files in full (no 30-line limit)
   - Identifies language-specific patterns:
     - Python: `__init__.py` nesting depth, decorator usage, FastAPI router pattern
     - TypeScript: barrel exports, React component structure, Next.js app router pattern
   - Returns `DeepPatternReport` with full snippets (up to 100 lines each)
   - Cleans up clone in `finally` block

2. `def detect_framework(repo_path: str) -> str | None`
   - Checks for framework markers: `fastapi`, `flask`, `django`, `react`, `next.js`, `express`, `tailwind`
   - Returns framework name or None

**New dataclass:**
```python

    language_patterns: list[Pattern]  # language-specific idioms

## Analysis: Official GitHub MCP Server

The official `@modelcontextprotocol/server-github` provides **17 lower-level GitHub API tools**:

| Category | Tools |
|----------|-------|
| Issues | `create_issue`, `list_issues`, `get_issue`, `update_issue`, `add_issue_comment` |
| Pull Requests | `create_pull_request`, `merge_pull_request`, `get_pull_request`, `get_pull_request_status`, `get_pull_request_comments`, `get_pull_request_reviews`, `get_pull_request_files`, `update_pull_request_branch`, `list_pull_requests` |
| Repos | `create_repository`, `fork_repository`, `search_repositories` |
| Files | `get_file_contents`, `create_or_update_file`, `push_files` |
| Code | `search_code`, `search_issues`, `search_users` |
| Branches | `create_branch` |
| Commits | `list_commits` |

**No overlap with repo-finder:**
- Official server is a **raw API wrapper** — does CRUD operations directly against GitHub
- repo-finder is a **decision/compression layer** — finds, ranks, inspects, extracts patterns
- Official `search_repositories` returns raw API results with no ranking
- Official `get_file_contents` reads one path at a time with no analysis
- Official has no concept of verdicts, quality scoring, pattern extraction, or comparison

**Complementary workflow:**
1. `repo_finder:find_repos_for_task` → identify candidate repos
2. `repo_finder:inspect_github_repo` → evaluate quality and structure
3. `github:get_file_contents` → drill into specific files of interest
4. `repo_finder:compare_github_repos` → choose the best option
5. `repo_finder:extract_patterns_from_repo` → get implementation patterns
6. `github:create_issue` / `github:create_pull_request` → act on findings

## Analysis: mcp-git-ingest (adhikasp/mcp-git-ingest)

**What it does:** 2 tools — `git_directory_structure` and `git_read_important_files`. Both clone the repo via GitPython, then operate on local files.

**Key patterns to borrow:**

| Pattern | Details |
|---------|---------|
| **GitPython local clone** | Faster and more reliable than API-only for deep file reading (no API rate limits for file-by-file access). Clones to temp dir, cleans up after. |
| **Deterministic caching** | Uses `sha256(repo_url)[:12]` for temp dir naming. Reuses clone if already present. |
| **Automatic cleanup** | Always cleans up cloned repos in `finally` blocks. Never leaks disk space. |
| **Minimal tool surface** | Only 2 focused tools. No feature creep. |
| **Tree formatting** | Visual Unicode tree representation (box-drawing chars). More human-readable than flat file list. |
| **Error resilience** | Graceful handling per-file: if one file can't be read, others still return. Returns error strings per file, not whole-operation failure. |

**Limitations relative to repo-finder:**
- No search capability — agent must already know the repo URL
- No ranking or quality assessment — no verdicts
- No pattern extraction — returns raw file contents
- No comparison — single-repo only
- No filtering — reads whatever file paths you give it
- Last push Jan 2025 — project appears inactive

**What to adopt in Phase 2:**
1. GitPython-based local clone for deep file access (no tree-sitter yet, but better than API-only 30-line previews)
2. Deterministic temp dir naming via repo URL hash
3. Automatic clone cleanup
4. Tree visualization for directory structure output
5. File-level error resilience (partial success on multi-file reads)

## Risks / Edge Cases

| Risk | Mitigation |
|------|-----------|
| GitHub rate limits exhausted quickly during testing | Cache aggressively. Use `@pytest.mark.integration` and skip by default. Set high TTLs during development. |
| Pattern extractor produces generic/low-quality output | Clearly document the MVP limitation. Focus extraction on README sections + file tree structure. Accept that this is a "decision aid" not a "code generator". |
| DuckDB in-memory cache lost on server restart | Acceptable for MVP. Document that `CACHE_DB_PATH` env var enables persistent cache. |
| Search API returns 0 results for valid queries | Ranker should handle empty lists gracefully. Return `FindReposResult(results=[], total_candidates_scored=0)`. |
| Private repos or 404 repos cause confusing errors | `get_repo_metadata()` returns clear `ToolError("Repository not found or is private")`. Never leak stack traces. |
| FastMCP version breaking changes | Pin to `fastmcp>=2.0.0,<3.0.0`. FastMCP v2 is the stable line with `@mcp.tool()` decorator. |
| httpx timeout on slow GitHub responses | Set 30s timeout on httpx client. Return `RuntimeError("GitHub API request timed out")` on timeout. |
| Ranker weights produce unexpected results | Make weights configurable via env vars in phase 2. For MVP, use hardcoded weights and document the formula. |
| README is enormous (e.g., documentation repos) | Truncate README to 10KB in `get_readme()`. Full content available only via explicit request. |
| Unicode/special chars in repo descriptions | httpx handles UTF-8 automatically. Ensure `fetch` calls don't double-encode. |
| GitPython clone large repos (>500MB) | Add `--depth=1` shallow clone. Set 60s timeout. Reject repos > 200MB with clear error. |
| Temp disk space exhaustion | Clean up after every clone. Deterministic naming means max N repos cached at once. |
| Clone of private repos | Requires different token scopes. Return clear ToolError. Document limitation. |
| Pattern library memory growth | Cap at 100 entries. Add LRU eviction. In-memory only for Phase 2. |
